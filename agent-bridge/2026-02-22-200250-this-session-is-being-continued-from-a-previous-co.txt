  What changed and why it will work now:

  1. EnsureUSDC(ctx, minUSDC) — runs automatically before every long trade:
    - If admin has no USDC trustline → creates one via ChangeTrust
    - If admin has insufficient USDC → sells XLM→USDC on SDEX to top up
    - Admin always has XLM (needed for Soroban fees), so this bootstrap always
  has fuel
  2. Flow for a user opening a 10 USDC long:
    - EnsureUSDC checks admin balance (e.g. 0 USDC, no trustline)
    - Creates USDC trustline
    - Sells ~10.1 XLM → gets ~10.1 USDC from SDEX
    - Then BuyXLM: sends 10 USDC → receives XLM (user's actual position)

  The bridge doesn't need the pool USDC directly — it uses XLM (which admin
  always has) to bootstrap, and the Soroban pool handles the accounting/PnL
  settlement side.

● Matching Engine Architecture

  The matching engine is a classic Central Limit Order Book (CLOB) running
  in-process in the Go bridge.

  PlaceOrder(order)
        │
        ▼
    OrderBook (per symbol: "XLM/USDC")
    ┌─────────────────────────────────────┐
    │  Bids (buy)  │  Asks (sell)         │
    │  sorted DESC │  sorted ASC          │
    │  [0.1010]    │  [0.1008] ← best ask │
    │  [0.1005]    │  [0.1009]            │
    │  [0.1000]    │  [0.1012]            │
    └─────────────────────────────────────┘
         │
         ▼ if bid >= best ask → FILL
    MatchResult { buyToken, sellToken, price, amount }
         │
         ▼
    processFill goroutine
    → Soroban.OpenPosition (on-chain)
    → LiquidationEngine.AddPosition

  Fill logic: price-time priority. When a new order crosses the spread (bid ≥
  ask), it matches against resting orders. Fills are returned synchronously; the
   on-chain call happens in a detached goroutine so the HTTP response isn't
  blocked.

  LiquidationEngine runs a ticker every 5s, computes unrealised loss for every
  monitored position against the live PriceSync mark price, and calls
  SettleTrade if loss ≥ 90% of collateral.

  PriceSync is a shared in-memory price map that the order book watcher writes
  to (from Horizon polling) and the liquidation engine reads from.

❯ when i deploy where will the things get stored 

● Good question — this is critical for production. There are three layers with
  very different persistence:

  ---
  What survives a bridge restart

  On Stellar blockchain (permanent)

  - AgentVault — every user's USDC balance, total pool balance
  - LeveragePool — open position records, locked collateral, free margin
  balances

  These live on-chain forever. No bridge or frontend involved.

  Browser localStorage (per device)

  - Bridge session token (uv-bridge-token)
  - Survives page refresh, dies if user clears browser storage

  ---
  What gets wiped on bridge restart (the problem)

  The bridge has zero persistent storage. Everything lives in Go memory:

  In-memory: Session token → Stellar address map
  Lost on restart: All users get "invalid token" errors
  ────────────────────────────────────────
  In-memory: Open positions (for liquidation)
  Lost on restart: Liquidation engine goes blind — can't liquidate anyone
  ────────────────────────────────────────
  In-memory: Order book orders
  Lost on restart: All resting limit orders vanish
  ────────────────────────────────────────
  In-memory: Mark prices
  Lost on restart: Price feed resets until watcher polls again

  ---
  What this means for deployment

  If you deploy the bridge to a VPS and it crashes or redeploys:

  1. Users' vault balances are safe (on-chain)
  2. Users' open leverage positions still exist on LeveragePool on-chain — but
  the bridge doesn't know about them, so it won't liquidate them and users can't
   close them through the UI
  3. Users need to reconnect their wallet to re-register their token

  ---
  The fix before production

  You need a persistent store for at minimum two things:

  1. Token → address map (already partially handled — users re-register on
  connect, but it breaks mid-session)

  2. Open positions — this is the dangerous one. Options:

  - Redis — fast, works well for session tokens and position state
  - PostgreSQL — if you want full trade history
  - On-chain only — on reconnect, query LeveragePool.get_position(user) to
  reconstruct bridge state from the chain itself (cleanest approach, no extra DB
   needed)

  The on-chain reconstruction approach is the most elegant: when a user
  reconnects, the bridge can call get_position(userAddr) and rebuild the
  Position struct from what's already on LeveragePool. That way the blockchain
  is your database.

❯ but what happens when we deploy this backend ?? 

● Good catch. It depends entirely on where you deploy.

  ---
  The problem

  Most cloud platforms use ephemeral filesystems — the disk is wiped on every
  deploy or container restart. bridge.db would be gone every time you push new
  code.

  Platform: VPS (DigitalOcean, EC2, Linode)
  Filesystem: Persistent disk
  bridge.db survives deploy?: Yes — just works
  ────────────────────────────────────────
  Platform: Railway / Render / Fly.io
  Filesystem: Ephemeral container
  bridge.db survives deploy?: No — wiped on redeploy
  ────────────────────────────────────────
  Platform: Docker without volume
  Filesystem: Ephemeral
  bridge.db survives deploy?: No
  ────────────────────────────────────────
  Platform: Docker with mounted volume
  Filesystem: Persistent
  bridge.db survives deploy?: Yes

  ---
  Fix for each scenario

  VPS (simplest)

  Run the binary directly on the server. bridge.db lives on the server disk.
  Survives restarts and redeploys. This is the easiest option for your scale.

  # on your VPS
  ./bridge-server   # bridge.db created next to it, stays forever

  Railway / Render / Fly.io

  These platforms offer persistent disk addons:
  - Railway → add a Volume, mount at /data, set DB_PATH=/data/bridge.db
  - Render → add a Disk, mount at /data, set DB_PATH=/data/bridge.db
  - Fly.io → fly volumes create, mount at /data, set DB_PATH=/data/bridge.db

  Docker

  # in your Dockerfile
  ENV DB_PATH=/data/bridge.db

  # run with:
  docker run -v bridge_data:/data ./bridge-server

  The volume bridge_data persists across container rebuilds.

  ---
  Honest recommendation

  For a testnet demo at this stage — use a VPS. Deploy to a $5/month
  DigitalOcean droplet, run the binary, done. No volume config, no infra
  complexity, bridge.db just lives on disk.

  For production later, switch to Turso (hosted SQLite, free tier, zero infra) —
   it's a drop-in replacement for modernc.org/sqlite that stores the database in
   the cloud instead of a local file.
